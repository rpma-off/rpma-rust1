## ЁЯФ┤ HIGH тАФ rusqlite Dependencies in Domain Layer (ADR-002)

**Rule:** SQL statements and `rusqlite` dependencies are strictly confined to the
`infrastructure/` and `src-tauri/src/db/` modules.

**Pattern of violation:** Domain models implement `FromSqlRow` (or `ToSql`) directly,
introducing a hard compile-time dependency on `rusqlite::Row` and friends into the
domain layer. This couples business entities to the persistence driver.

### Affected files (12 domain models, ~22 import violations)

| Domain | File | Violation |
|--------|------|-----------|
| calendar | `domain/models/calendar.rs` | `use crate::db::FromSqlRow; use rusqlite::Row;` |
| calendar | `domain/models/calendar_event.rs` | `use crate::db::FromSqlRow; use rusqlite::Row;` |
| clients | `domain/models/client.rs` | `use crate::db::FromSqlRow; use rusqlite::Row;` |
| documents | `domain/models/photo.rs` | `use crate::db::FromSqlRow; use rusqlite::{тАж, ToSql};` |
| interventions | `domain/models/intervention.rs` | `use rusqlite::{Result as SqliteResult, Row};` |
| interventions | `domain/models/step.rs` | `use crate::db::FromSqlRow; use rusqlite::{тАж, Row};` |
| inventory | `domain/models/material.rs` | `use crate::db::FromSqlRow; use rusqlite::Row;` |
| notifications | `domain/models/message.rs` | `use crate::db::FromSqlRow; use rusqlite::Row;` |
| notifications | `domain/models/notification.rs` | `use crate::db::FromSqlRow; use rusqlite::{тАж, Row};` |
| organizations | `domain/models/organization.rs` | `use crate::db::FromSqlRow; use rusqlite::Row;` |
| organizations | `domain/models/settings.rs` | `use rusqlite::Row;` |
| quotes | `domain/models/quote.rs` | `use crate::db::FromSqlRow; use rusqlite::Row;` |

> **Note:** `tasks/domain/models/task.rs` has been **fixed** (see Patch P-1 below).

### Fix pattern (established by tasks domain pilot)

1. Create `infrastructure/<domain>_row_mapping.rs` with the `impl FromSqlRow for <Model>` blocks.
2. Add `pub(crate) mod <domain>_row_mapping;` to `infrastructure/mod.rs`.
3. Remove `use rusqlite::Row`, `use crate::db::FromSqlRow`, and any `impl FromSqlRow`/`impl ToSql` from the domain model file.
4. Run `cargo check --lib` to confirm no regressions.

**Effort per domain:** ~30 min each. Low risk тАФ pure code movement, no logic changes.

## ЁЯЯб MEDIUM тАФ Cross-Domain Orchestration in IPC Handler (ADR-005)

**Rule:** IPC handlers should authenticate, establish request context, map
inputs/outputs, and delegate. Complex orchestration belongs in the application layer.

### Affected file

**`src-tauri/src/domains/quotes/ipc/quote.rs` тАФ `quote_convert_to_task` handler (lines 730тАУ848)**

The handler:
1. Fetches the quote from the quotes domain.
2. Constructs a `crate::domains::tasks::domain::models::task::CreateTaskRequest` directly.
3. Calls `state.task_service.create_task_async(...)` (cross-domain service call).
4. Then calls `facade.convert_to_task(...)` to update quote status.

Steps 2тАУ4 are **orchestration** and cross-domain logic that belongs in the quotes
`application/` layer, not the IPC handler.

### Proposed fix

Move the body of `quote_convert_to_task` (excluding authentication and request
deserialization) into `QuoteService::orchestrate_quote_to_task_conversion(...)` in
`src-tauri/src/domains/quotes/application/quote_service.rs`. The `QuoteService`
will need a reference to the `TaskService` injected at construction time (or via
an application-level orchestration service).

**Effort:** ~2h. Medium risk тАФ requires updating constructor injection and test setup.

## ЁЯЯв LOW тАФ Accepted Cross-Domain Imports in Application Layer (ADR-001)

These cross-domain dependencies are **intentional and documented** in the
allowlist in `scripts/backend-architecture-check.js`.

| Source Domain | Target Domain | Layer | Justification |
|---------------|--------------|-------|---------------|
| documents | interventions | application | `report_view_model.rs` and `report_pdf.rs` need `InterventionType`, `StepStatus`, `StepType` to render PDF reports |
| documents | clients | application | `report_view_model.rs` uses `CustomerType` for report metadata |
| reports | documents | application | `report_service.rs` delegates PDF generation to `documents::application::report_export` |

**Action:** No code change needed. The allowlist in the enforcement script documents the exception.
